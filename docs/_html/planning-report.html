1. 제품 정의

Paperclip Company OS는 사용자가 혼자 운영하지만, 내부적으로는 여러 부서가 있는 회사처럼 작동하는 AI 미디어커머스 운영 시스템이다.

이 시스템의 핵심 구조는 다음과 같다.

Hermes는 플랫폼 내부의 시장조사 부서다.
Paperclip은 플랫폼 전체를 감싸는 경영·판단·승인 레이어다.
Content Engine은 실제 콘텐츠를 만든다.
Channel Profiles는 블로그, 클립, 인스타그램, Threads, X의 문법을 적용한다.
ShoppingConnect Engine은 수익 동선을 설계한다.
Compliance Engine은 광고표시, 가격, 출처, 과장 표현을 막는다.
Analytics / CFO Engine은 성과와 수익을 기록한다.

한 줄 정의:

Hermes가 시장 기회를 찾고, Paperclip이 오늘의 전략과 작업을 결정하고, 부서형 AI Agent들이 블로그·홈피드·쇼핑커넥트·클립·SNS·검수·성과분석을 수행하며, 사용자는 최종 오너로 승인하는 1인 AI 미디어커머스 회사 운영체제.

이 제품은 단순한 "AI 글쓰기 도구"가 아니다.

시장 탐색
→ 기회 발굴
→ 홈피드/검색/수익성 점수화
→ 오늘의 콘텐츠 결정
→ 블로그 원문 생성
→ 쇼핑커넥트 수익 동선 설계
→ 네이버 클립/SNS 변환
→ 컴플라이언스 검수
→ 사용자 승인
→ 수동 게시
→ 성과 기록
→ 다음 전략 개선

을 수행하는 1인 회사 운영체제다.

2. 정책·플랫폼 전제

  • 네이버 홈피드는 검색홈 하단 추천 콘텐츠 피드. 로그인 사용자의 활동·관심사 기반 개인화 노출. → Paperclip은 관심사 기반 홈피드 패키징을 함께 설계한다.
  • 홈피드 세부 랭킹 로직은 외부에서 알 수 없음 → "노출 보장"이 아니라 홈피드 적합도 점수를 내부 계산.
  • 네이버 블로그 글쓰기 API는 2020-05-06 종료 → MVP에서 자동 발행 제외, Markdown/HTML/복사용 Export + 수동 게시.
  • 서치어드바이저 가이드: 명확한 제목·본문 구조·비교표·FAQ·자연스러운 키워드 회수.
  • 쇼핑커넥트: 판매자가 등록한 상품만 링크 발급 가능. 직접 실적(본인 링크로 해당 상품 직접 조회 후 주문, 상품별 수수료) vs 간접 실적(24시간 내 다른 상품 구매, 기본 1.8%). 유입 유효 24시간, 집계 유효 7일 → 특정 상품을 직접 클릭하게 만드는 구조에 집중.
  • 네이버 클립에서도 쇼핑커넥트 스티커 사용 가능, 대가성 문구 필요. 블로그·클립·Instagram·YouTube 중 1개 채널만 있으면 구독자 수 제한 없이 이용 가능.
  • 클립 정보 태그 → 네이버 앱 홈피드·검색·플레이스·지도·플러스스토어 노출 가능 → 블로그 원문 1개당 클립 대본 + 정보태그/쇼핑태그 메모 함께 생성.
  • Instagram/Facebook은 AI 랭킹, Threads는 텍스트 대화형, X For You는 상호작용 신호 기반 → SNS 변환은 단순 요약이 아니라 채널별 행동 목표에 맞게 다시 쓴다.

참고 링크: 네이버 고객센터(홈피드/쇼핑커넥트), NAVER D2, 네이버 개발자센터, 서치어드바이저, mkt.naver.com 클립 프로필, Meta/X 공식 문서 (원본 v0.6 각주 참조)

7.5 HomeFeed Optimizer

노출 보장 엔진이 아니라 홈피드 적합도 내부 판단 엔진.

(v0.7) 점수는 4축으로 단순화:

homefeed_score  : 관심사 명확성 + 후킹 강도 + 첫 화면 품질 - 낚시 위험 (AI 루브릭 채점 + 근거 문장)
search_score    : 검색 수요 + 키워드 구조 적합성 (AI 루브릭 채점 + 근거 문장)
revenue_score   : 문제-상품 적합 + 직접 클릭 의도 + 수수료 (AI 루브릭 채점 + 근거 문장)
risk_score      : 컴플라이언스 위험 + 광고 피로도 (낮을수록 좋음)
원본 v0.6의 11변수 가중치 공식은 성과 데이터가 쌓인 후(Phase 4+) 정교화 시 참고용으로 보존한다. 각 점수에는 반드시 "왜 이 점수인지" 근거 문장을 함께 출력한다.

홈판형 제목 공식 (유형 7종):

문제 공감형: 자취방 냄새, 청소보다 습기 문제일 수 있어요
실수 지적형: 방향제부터 사면 냄새 해결이 늦을 수 있어요
비교 선택형: 원룸은 제습제와 미니 제습기 중 뭐가 나을까
시즌 긴급형: 장마 시작 전에 봐야 할 자취방 습기템
체크리스트형: 장마 전 자취방에서 확인할 5가지
반전형: 자취방 냄새는 방향제보다 제습이 먼저입니다
구역분리형: 옷장·침대 밑·신발장은 제습템이 달라야 해요

risk_score     : 걸릴 게 있는가 (컴플라이언스 위험, 광고 피로도 — 낮을수록 좋음)

발행 가능 기준 (Publish Readiness):

- Compliance Gate 통과 (high risk 0건)
- homefeed_score 또는 search_score ≥ 70
- revenue_score ≥ 65 (수익형 글인 경우)
- 사용자 최종 승인
v0.6의 11변수 opportunityscore / publishreadiness / companygrowthscore 공식은 성과 데이터 축적 후 정교화 시 참고용으로 보존 (부록 성격).

핵심 설계 원칙

"하루 30분, 입력 최소화"

모든 UI 설계의 1순위는 사용자가 30분 안에 완료할 수 있도록 입력 필드를 최소화하는 것입니다.

입력 최소화의 예:
❌ 상품 등록: 상품명, URL, 가격, 수수료, 주의사항, 카테고리, 태그, 설명 (8개 필드)
✓  상품 등록: URL 붙여넣기 (자동 크롤링) / 확인일 입력 (1개 필드)

❌ 성과 기록: 조회수, 클릭, 좋아요, 댓글, 공유, 저장, 직접 수익, 간접 수익 (8개)
✓  성과 기록: 조회수, 클릭 (2개 필드) + 수익은 선택

❌ 주제 선택: 상세 분석 + 카테고리 선택 + 키워드 입력 + 선호도 설정
✓  주제 선택: [선택] 버튼 1개 (Hermes가 모든 분석 완료)

POST 라우트 (데이터 생성/수정)

// app/api/hq/decisions/route.ts

import { NextRequest, NextResponse } from "next/server";
import { auth } from "@/lib/auth";
import { db } from "@/lib/db";
import { validateDecisionRequest } from "@/lib/validators";
import { PostHermesDecisionRequest } from "@/types/api";

export async function POST(
  req: NextRequest
): Promise<NextResponse> {
  const requestId = crypto.randomUUID();

  try {
    const session = await auth();
    if (!session) return unauthorized(requestId);

    // 요청 검증
    const body = await req.json();
    const validation = validateDecisionRequest(body);
    if (!validation.valid) {
      return badRequest(validation.errors, requestId);
    }

    const request = body as PostHermesDecisionRequest;

    // 트랜잭션 처리
    const result = await db.$transaction(async (tx) => {
      // 1. OpportunityMemo 조회
      const memo = await tx.opportunity_memos.findUnique({
        where: { id: request.opportunity_memo_id },
      });
      if (!memo) throw new Error("Memo not found");

      // 2. ContentPackage 생성
      const contentPackage = await tx.content_packages.create({
        data: {
          topic_id: memo.id,
          status: "assigned",
          publish_readiness: "not_ready",
        },
      });

      // 3. PaperclipDecision 기록
      const decision = await tx.paperclip_decisions.create({
        data: {
          opportunity_memo_id: request.opportunity_memo_id,
          decision: request.decision,
          reason_json: request.reason || [],
        },
      });

      return { contentPackage, decision };
    });

    return NextResponse.json({
      success: true,
      data: result,
      error: null,
      timestamp: new Date().toISOString(),
      request_id: requestId,
    });
  } catch (error) {
    // 에러 처리
    return serverError(error, requestId);
  }
}

// 헬퍼 함수들
function unauthorized(requestId: string) {
  return NextResponse.json(
    {
      success: false,
      data: null,
      error: { code: "UNAUTHORIZED", message: "Not authenticated" },
      timestamp: new Date().toISOString(),
      request_id: requestId,
    },
    { status: 401 }
  );
}

function badRequest(errors: Record<string, string>, requestId: string) {
  return NextResponse.json(
    {
      success: false,
      data: null,
      error: {
        code: "VALIDATION_ERROR",
        message: "Invalid request",
        details: errors,
      },
      timestamp: new Date().toISOString(),
      request_id: requestId,
    },
    { status: 400 }
  );
}

function serverError(error: unknown, requestId: string) {
  return NextResponse.json(
    {
      success: false,
      data: null,
      error: {
        code: "INTERNAL_SERVER_ERROR",
        message: error instanceof Error ? error.message : "Unknown error",
      },
      timestamp: new Date().toISOString(),
      request_id: requestId,
    },
    { status: 500 }
  );
}

08-business-model.md: 비즈니스 모델

Paperclip Company OS v0.7 "1인 AI 미디어커머스 회사" 수익 구조 기준일: 2026-07-02

핵심 원칙

Paperclip Company OS는 SaaS가 아니다.

  • 구독료 없음
  • 결제 시스템 없음
  • 1인 개인용 운영 시스템
  • 사용자가 직접 네이버 쇼핑커넥트 수익화

1. 수익 흐름

사용자가 Paperclip으로 콘텐츠 생성
    ↓
블로그 + 쇼핑커넥트 링크 게시
    ↓
독자가 링크 클릭
    ↓
네이버 쇼핑에서 상품 구매
    ↓
사용자가 쇼핑커넥트 수수료 수익 (직접 실적)
또는
사용자가 다른 상품 수익 (간접 실적, 24시간 내)
    ↓
성과 기록 → Company Memory 학습
→ 다음 콘텐츠 성공률 개선

2. 네이버 쇼핑커넥트 수익 구조

직접 실적 (Direct Performance)

사용자의 쇼핑커넥트 링크 클릭
  ↓
"이 상품을 구매합니다" (구매 확정)
  ↓
네이버 쇼핑커넥트에서 판매자에게 주문 전달
  ↓
판매자가 판매 건수 기준으로 수수료 발생
  ↓
사용자가 쇼핑커넥트 대시보드에서 "직접 실적" 카운트 확인
  ↓
수수료: 상품별 2~3% 수준 (네이버 정책에 따라 변동)

예시:

  • 제습제 8,900원 × 클릭 10명 × 1명 구매 × 2% = 178원
  • 미니제습기 45,000원 × 클릭 20명 × 2명 구매 × 2% = 1,800원

수익 집계

월간 수익 요약:
  SELECT 
    DATE_TRUNC('month', ordered_at) as month,
    SUM(amount) as total_revenue,
    revenue_type,
    COUNT(*) as transaction_count

  GROUP BY month, revenue_type;

  SELECT 
    cp.topic_id,
    p.post_url,
    SUM(rl.amount) as total_revenue,
    pl.views,
    pl.clicks,
    (SUM(rl.amount) / pl.views * 100) as revenue_per_view_pct
  FROM content_packages cp
  JOIN performance_logs pl ON cp.id = pl.content_package_id
  LEFT JOIN revenue_logs rl ON ... (매핑 필요)
  GROUP BY cp.id, pl.id;

2. 다시 정리한 한 줄 문제 (Real Problem)

혼자서 매일 수익형 콘텐츠를 기획 → 제작 → 검수 → 게시 → 회고까지
모두 돌리려면 회사 수준의 운영 구조가 필요한데,
그걸 사람 손으로 다 하면 지속이 안 된다.

이 문제에 도달한 이유:

문제 분석 (왜 "글쓰기 도구"로는 부족한가)

사용자가 하던 일:

1. 아침 (10분)
   "뭘 쓸까?" 고민
   ↓
2. 오전 (30분)
   블로그 글 작성 (제목, 구조, 본문)
   ↓
3. 낮 (15분)
   쇼핑커넥트 링크 찾고 배치
   ↓
4. 오후 (20분)
   SNS 올리기 (Instagram, Threads, X)
   ↓
5. 저녁 (10분)
   성과 기록, 다음 전략 고민
   ↓
6. 이틀에 한 번
   "다음엔 뭘 하지?" 전략 재수립

결과: 지속 불가능 (월 2~3글만)

"글쓰기 도구"로 해결 안 되는 부분:

1. 기획: "뭘 쓸까" 고민이 자동화되나? NO
   → AI 글쓰기 도구는 "이미 결정된 주제"만 쓸 수 있음
   → 주제 선택 자체가 가장 어려운 부분

2. 전략: "홈피드 제목" vs "검색 제목" 두 개를 어떻게 쓸 것인가?
   → 같은 글 2개 버전? 아니면?
   → AI 도구는 순수 글만 씀

3. 수익화: 쇼핑커넥트 링크를 어디에 어떻게 배치할 것인가?
   → "링크 여기 추가" 수준이 아니라, "CTA 전략"이 필요
   → 글 완성 후 별도 작업

4. 검수: 대가성 문구는 있나? 가격 기준일은 있나?
   → AI가 실수하는 부분
   → 사용자가 수동으로 다시 확인

5. 출판: 블로그 / 클립 / SNS 각각 다른 형식
   → 글을 복사해서 SNS에 붙일 수 없음
   → 채널별 "변환"이 필요

6. 학습: "지난번 잘 된 제목은?" → 패턴이 자동으로 반영되나?
   → NO, 사용자가 기억했다가 "다음엔 이렇게 쓸게"라고 해야 함

→ 결론: "글쓰기"만으로는 월 1글도 어렵다.
          필요한 건 "회사처럼 돌아가는 시스템"이다.

3. 다시 정리한 한 줄 (JTBD 형태로, 라벨 없이)

"매일 아침 30분,
 오늘 만들 수익형 콘텐츠 1개를 고르고,

1. 매일 기회가 자동으로 떠야 하고 (Hermes)

5. 게시만 수동으로 하면 (Export + Manual Publish)

- 수익이 ₩500k에 가까운가?
- 사용자가 "이 시스템이 없으면 못 한다"고 느끼는가?

→ 위 3가지 모두 "YES"면 문제 해결 증명

7. 실패 시나리오 (회피해야 할 것)

이렇게 되면 안 된다

시나리오 1: "구조는 좋은데 시간이 여전히 많이 걸려"
  증상: 사용자가 "아침 1시간, 저녁 1시간" 투자 필요
  원인: 입력 필드 너무 많음 (상품 5개 세부정보 입력 등)
  해결: "URL 붙여넣기"까지 최소화
  
시나리오 2: "품질이 떨어져서 쓸 수 없어"
  증상: 콘텐츠 패키지가 나와도 사용자가 "다시 써야 함"
  원인: AI 모델이 부족하거나 프롬프트가 부정확
  해결: Claude / GPT-4 수준 필수. 테스트 후 출시
  
시나리오 3: "지난번 패턴이 다음엔 안 반영되더라"
  증상: Company Memory가 계속 실패
  원인: 학습 로직이 너무 복잡하거나 버그
  해결: 간단한 규칙부터 시작 (후킹 유형별 조회수 평균만)
  
시나리오 4: "매번 수동 정정이 필요해"
  증상: Compliance Gate가 계속 실패 뜸
  원인: 규칙이 너무 까다롭거나 잘못 설정됨
  해결: High Risk 0건, Medium은 사용자 수동 확인만
  
시나리오 5: "결국 SNS 자동 발행을 원하는데..."
  증상: 사용자가 "게시도 자동으로 해줘"라고 함
  원인: 자동 발행 제약 설명이 부족
  해결: "정책 리스크" 명확히 설명. Phase 2+에서 검토

8. 제품 vs 솔루션

"이건 제품인가, 솔루션인가?"

제품 (Product):
  = "블로그 글쓰기 AI도구"
  사용자 입장: "편한 도구네" (하지만 혼자여야 함)

솔루션 (Solution):
  = "1인 회사 운영 OS"
  사용자 입장: "내가 경영실장, 결정권자가 됐다"

Paperclip은?
  → "솔루션"이다.
  
이유:
1. AI 도구가 아니라, "시스템 구조"를 제공
2. 사용자는 "쓰는 사람"이 아니라 "결정하는 경영자"
3. 성공의 핵심은 "좋은 글"이 아니라 "좋은 의사결정"
4. 가능성: 사용자가 "다른 채널" (쿠팡, 11번가) 추가 가능

→ 이를 위해:
   - 채널별 Agent 분리 X (Profile만 바꿈)
   - 모든 구조를 "의사결정 중심"으로 설계
   - 승인은 항상 "사용자 오너"가 함
   - AI는 "제안만" 함

9. 최종 가설 (Theory of Change)

우리가 믿는 것:

1. 문제: "혼자는 지속 불가능"
   증거: 사용자가 월 2~3글에서 6개월 정체됨

2. 원인: "회사처럼 분업할 수 없음"
   증거: 기획/제작/검수/출판/분석을 모두 1인이 함

3. 솔루션: "AI + 구조 = 시스템"
   증거: Hermes(조사) + Paperclip(결정) + Agents(실행) + User(승인)

4. 예상 효과: "30분 투자로 월 4글, ₩500k"
   검증: Month 3까지 데이터 수집 후 판단

5. 장기 가능성: "1인이 "회사"처럼 운영"
   비전: 다른 채널 추가 / SaaS화 / 팀 확대

→ 이 가설이 맞다면, Paperclip은 단순 도구가 아니라 "패러다임 전환"이다.
