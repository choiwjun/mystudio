## 1. 참여자 및 점수

| 리뷰어 | 초기 점수 | 교차검증 후 | 변동 사유 |
|--------|-----------|-------------|-----------|
| CTO | 5 | 4 | 타 리뷰어 지적(승인 상태 부재, 컴플라이언스 무기록 오버라이드 등)이 자신의 상태 머신·인프라 지적과 결합해 구조적 결함임을 확인 |

## 2. 합의 사항 (3인 전원 동의)

교차검증에서 세 리뷰어가 모두 동의(agree, 또는 해당 범위에 한정해 명시적 동의)한 항목만 기재.

### 2.1 P0 — 구현 착수 전 해소 필수

1. **Cron 엔드포인트(`POST /api/hermes/scan`) 무인증 + 멱등성 부재.** CRON_SECRET 검증과 실행 단위 멱등키가 없으면 외부인이 AI 비용 폭주(경제적 DoS)를 유발할 수 있고, 정당한 재시도만으로도 opportunity_memos가 중복 생성되어 "아침 3~5카드, 5초 판단" 핵심 가치가 직접 훼손됨. 인증(보안)과 멱등성(정합성)은 별개 수정이며 둘 다 저비용. (CTO·Security 독립 발견, UX 동의)
2. **AI 비용 상한/서킷브레이커 부재.** cost_logs는 기록만 함. 소프트 임계(알림) → 하드 캡(차단) 2단계 + 진행 중 파이프라인 완주 보장(soft-stop)으로 설계해야 하며, "차단된 상태"의 화면/헤더 알림도 함께 정의 필요. (CTO·Security 독립 발견, UX 동의)
3. **상태 머신 단일 정본 부재.** status가 4개 테이블에 String으로 분산되고 전이 검증·이력이 없음. Security 관점에서는 임의 status 갱신으로 Compliance Gate(export 차단)를 우회 가능한 통제 공백이고, UX 관점에서는 "내 글이 어디 갔지" 버그의 구조적 원인. 승인 상태 부재·칸반 매핑 부재·헤더 상태 불일치가 전부 이 문제의 파생물이므로 **정본 확정이 다른 수정들보다 선행**되어야 함.
4. **승인(Approval) 개념의 화면-상태머신-DB 불일치.** 06-screens에는 [승인][반려]가 3곳 존재하나 03-user-flow 상태 전이도와 DB에는 승인 상태가 0건. 불일치 자체는 전원이 결함으로 확인 — 단 해소 방향(상태 추가 vs 화면 삭제)은 미합의(§4-1 참조).
5. **Stored XSS 미대응.** Draft.bodyHtml(AI 출력)·raw_items(제3자 블로그)가 sanitization/CSP 없이 렌더링됨(문서 전체에 관련 언급 0건). AI 출력과 외부 수집 콘텐츠는 신뢰 경계 밖 데이터로 취급해야 함.
6. **장시간 AI 파이프라인 vs Vercel serverless 타임아웃 충돌.** POST /generate 계열이 전부 동기 요청/응답이며 202+폴링/SSE, 중간 실패 재개, 진행률 UI가 모두 부재. "비동기 실행 모델 결정 + 그에 따른 대기/재개 UX 정의"가 P0라는 데 전원 동의 — 구체 수단(잡 큐 vs 중간해)은 미합의(§4-2 참조).
7. **인증 설계 모순.** TRD §10 'Bearer Token' vs 06-tasks 'NextAuth 세션'이 상충. 어느 쪽이든 **하나로 확정하는 결정 자체가 P0**이며, 최악은 두 방식이 공존한 채 절반만 구현되는 것.

### 2.2 P1 — 저비용·고효과 합의 수정

8. **Compliance [무시]의 무기록 오버라이드.** 감사 기록 없는 원클릭 소거는 Compliance Gate의 존재 가치(사후 방어 가능성)를 0으로 만듦. override 이력(누가·언제·어떤 룰·왜) 기록은 전원 동의 — 마찰 수준은 위험도 차등화(Low 원클릭+자동기록, High 사유 필수)로 절충.
9. **에러/경고 토스트 4초 자동 소멸.** 에러의 휘발은 UX 안티패턴이자 보안 이벤트 탐지 실패. 심각도 기반 분리(정보성=자동 소멸, 에러/컴플라이언스=수동 해제+재확인 경로) 필요.
10. **`/api/products/import`의 SSRF 무방비 + Won't 스코프 자기모순.** allowlist·내부 IP 차단·리다이렉트 제한이 전무하고, Won't로 제외한 '쇼핑커넥트 자동 스크래핑'과 사실상 동일 행위라 Naver ToS와 충돌.
11. **error_logs.context의 request_body 원문 저장.** 로그인 실패 시 비밀번호 평문이 90~180일 잔존. 거부목록 마스킹(password/token/apiKey) + 로그인 라우트 body 로깅 제외가 저비용 해법.
12. **규칙 기반 컴플라이언스의 LLM 판정 override 원칙 + AI 출력 JSON 스키마(zod) 검증 명문화.** Prompt injection 위협의 심각도 평가는 갈렸으나, 이 두 가지 최소선은 전원 동의(파이프라인 안정성에도 필수).
13. **보안 항목의 derived gates/acceptance 편입.** 현재 보안 태스크·게이트 0개. 풀 위협모델은 과잉이라는 데도 전원 동의 — P0 체크리스트 수준(CRON_SECRET, sanitization, 민감 필드 마스킹, npm audit, env 시크릿)이 적정선.
14. **미구현 사이드바 메뉴(7부서×20여 항목 vs 화면 5개)는 Phase 1에서 숨김.** 스텁 생성은 팬텀 스코프(CTO)와 검증 빠진 라우트 양산(Security)을 유발하므로 '숨김 > 스텁'에 전원 수렴.
15. **성능 목표 'Blog Draft <10초'는 그대로 두면 게이트 무력화.** 반드시 실패하는 게이트는 '게이트를 끄는 문화'를 만들어 Compliance Gate까지 오염(normalization of deviance). 재정의 필요 — 방식은 §3-11 참조.
16. **빈 프로필/Day 1 상태의 파이프라인 가드 부재.** company_profile 미설정 시 AI가 근거 없는 가격·효능 문구를 생성할 수 있고 Compliance Gate의 fail-open/closed 동작이 미정의. 풀 온보딩 위저드가 아닌 "미설정 시 생성 차단 + 설정 유도" 수준의 게이트로 충분하다는 데도 수렴.
17. **헤더 상태 색상 문법 위반 + 상태 개수 문서 불일치.** 정상 아침 상태(Focus)가 항상 빨강이면 경보 피로로 진짜 위험 신호(검수 실패)가 배경 소음화 — 표시광고법 게이트의 실효성이 UI 레이어에서 무너지는 경로. 상태 개수(00은 5개, 03/06은 4개) 드리프트도 수정 필요.
18. **문서 간 드리프트의 시스템성.** 승인 상태, 상태 개수, compliance_rules vs policy_rules, 보존 정책(90일 삭제 vs 무제한), 탭 명칭 충돌, Phase 2 편집 탭 유실 — 개별 오타가 아니라 문서 간 diff 검증 프로세스 부재의 증상이라는 진단에 전원 수렴.

## 3. 주요 개선 권고 (교차검증에서 정제된 실행안)

1. **상태 머신 정본 확정을 최우선 순서로.** 상태 축소(14개→8개 수준, UX) → 정본 테이블/컬럼 확정 + Prisma enum + 단일 전이 함수(서버측 검증) + 전이 이력 테이블 → 칸반 매핑·헤더 상태를 그 파생 함수로 도출. 역순(매핑 먼저)이면 재작업. UI 진입점이 몇 개든 단일 도메인 서비스를 경유하도록 강제(검증 중복 구현 방지, Security).
2. **비동기 파이프라인과 서킷브레이커를 하나의 아키텍처 결정으로 묶기.** 비용 차단만 단독 추가하면 동기 파이프라인 중간에서 죽어 데이터 유실이라는 새 장애 모드를 만듦(CTO). 단계 경계에서 예산 체크 후 soft-stop. 잡 큐 도입 시 웹훅 서명 검증 + 재시도 멱등성을 acceptance에 포함(Security).
3. **멱등키 단위는 '일자'가 아니라 '트리거 실행 ID'.** 일자별 unique는 예외 흐름 2의 [새로 스캔] 수동 재스캔 UX와 충돌해 기능 하나를 죽임(UX가 CTO 제안을 정정, CTO 원안 그대로 구현 금지).
4. **콘텐츠 정본을 markdown으로 유지하고 HTML은 Export 경계에서만 생성.** bodyHtml이라는 XSS 위험 표면 자체를 축소(CTO). sanitization은 이원화 — 미리보기용(DOMPurify 엄격) / Export 산출물용(네이버 에디터 호환 태그 allowlist, 콘텐츠 품질 태스크로도 등록)(UX). faq 필드 타입(JSON vs 마크다운) 확정이 XSS 대응 설계의 전제 조건(Security).
5. **인증 단일화 + 계정 보호의 적정선.** 단일 사용자 조건에서는 NextAuth 풀 구성보다 env 기반 단일 자격증명 + middleware rate limit이 공격 표면과 코드 모두 작을 수 있음(CTO). 단 공격적 lockout은 유일한 사용자가 자기 도구에서 잠기는 자해이므로 rate limit만 도입하고 lockout은 신중히(UX). 쿠키 세션 채택 시 상태 변경 API의 CSRF 대책과 세션 만료 시 에디터 자동 저장 손실 방지(만료 경고+로컬 보존+복원)를 세트로.
6. **TRD에 '데이터 접근 경계' 섹션 신설.** RLS 사용 여부, pgBouncer/DATABASE_URL vs DIRECT_URL, anon key 노출 정책, Storage 버킷 접근 제어를 한 번에 확정. 단일 사용자+Prisma 직결에서는 'RLS 미사용 + 전 접근 서버 경유 + 버킷 비공개+서명 URL'이 오히려 더 단순·안전할 수 있음 — 어떤 경계를 선택했는지 명문화하지 않은 것이 진짜 결함(CTO·Security 수렴).
7. **Company Memory 최소 실효선.** 알고리즘 전체 설계는 구현 단계로 미루되, ① pattern 자유 텍스트를 구조화 필드+enum 태그로(집계 가능성과 prompt injection 면적 축소를 동시 해결, Security), ② 표본 N<5 시 추천 억제/신뢰도 배지, ③ 패턴이 실제 추천에 쓰였을 때 화면 표시 — 이 3개는 기획 단계에서 확정(UX). "정직한 '배우는 중'이 가짜 지능보다 낫다."
8. **성과 기록 리마인드는 cron 알림 대신 '다음 접속 시 미기록 배지'.** cron 0개 추가로 동일 효과(CTO), 자동 수집(스크래핑)으로의 선회는 ToS 문제 재발이므로 금지(Security).
9. **성능 게이트 재정의는 수치 완화가 아니라 측정 대상 변경.** '생성 시간 <N초' 대신 '시작 피드백 <1초 + 단계별 진행 가시성 + 완료 알림 도달'(UX), 비용/호출량 기준선과 세트로 재설정(Security). derived gates(MG-008-1) 연쇄 수정 포함.
10. **products/import에 폴백 UX 필수.** allowlist 차단 시 "지원하지 않는 주소예요. 직접 입력하시겠어요?" 수준의 회복 가능한 에러 + 수동 입력 폼 폴백 — 보안 조치가 UX 후퇴로 체감되지 않게(UX).
11. **[되돌림] 버튼은 현 스키마로 구현 불가능.** 풀 리비전 테이블은 과잉 — AI 생성 직후 고정되는 originalBody 스냅샷 컬럼 1개로 '원본 되돌림'만 지원, 자동 저장 유지 시 [저장][취소]는 제거(CTO). 이력이 생기면 보존 정책 표 갱신 동반(Security).
12. **zod 검증의 단계적 적용 우선순위.** ① AI 출력 JSON(최대 수익 — LLM은 스키마를 확률적으로만 지킴) → ② 상태 변경·외부 데이터 유입 엔드포인트(import/scan/apply-fixes/approve) → ③ 일반 입력 → Json 컬럼 내부 구조는 후순위. 전 필드 P0는 과잉(CTO·Security 수렴).
13. **on_hold 재검토 경로.** 전용 화면 신설 대신 Hermes Desk 필터 탭 추가가 적정 비용(CTO, UX 지적 수용).
14. **모바일 우측 패널 동선 정의.** 승인처럼 게이트 역할을 하는 액션은 모든 폼팩터에서 접근 가능해야 통제로서 유효(불편한 통제는 우회됨, Security). 키보드 조작성·reduced-motion은 'MVP a11y 제외 항목'으로 명시적 문서화(침묵이 아니라 선택으로).
15. **터치 타깃은 반응형 분기로 해소.** 데스크톱 28px은 WCAG 2.5.8 AA(24px) 통과 — 전면 44px 대신 모바일 뷰 한정 44px 강제 + [폐기] 등 파괴적 액션에만 확인 단계 추가(CTO·Security 절충, 카드 밀도 목표 보존).

## 4. 미합의 쟁점 (사용자 결정 필요)

### 4-1. 승인 워크플로: 상태 추가 vs 화면 삭제
- **Security**: approved/rejected 상태 + 승인자·시각 기록을 상태 머신에 정식 편입. Owner 승인은 시스템의 유일한 인적 통제이며, 표시광고법 리스크 제품에서 "언제 무엇을 확인하고 내보냈나"의 기록 자체가 방어 자료.
- **CTO**: 1인 도구에서 소유자=승인자이므로 Owner Approval Queue 화면을 삭제하고 compliance_checked→exported 직행을 정본으로 확정하는 것이 가장 싼 해법일 수 있음. 승인 워크플로 추가는 매일 클릭 1회의 순수 마찰.
- **UX**: 어느 쪽이든 3개 화면과 상태 머신의 불일치 해소가 선결 — 방향 자체는 제품 결정.

### 4-2. 비동기 실행 수단: 잡 큐 vs 경량 중간해
- **CTO**: Inngest/QStash/Supabase Queues 등 잡 큐 도입이 P0. 중간 실패 재개(resume)까지 커버해야 함.
- **UX**: 풀 오케스트레이션은 eventual consistency로 인한 새 UX 복잡성(칸반 미반영, 완료 알림 채널 신설)을 만듦. Vercel Pro maxDuration + 클라이언트 단계별 순차 호출 같은 더 단순한 중간해 검토 — "잡 큐가 P0"가 아니라 "실행 모델 결정이 P0".
- **Security**: 큐 도입 시 웹훅 서명 검증·시크릿 증가라는 새 공격면 관리 비용 동반. 미도입 시 타임아웃 중단으로 미검수 Draft가 남는 무결성 문제.

### 4-3. 멀티테넌시: owner_id 지금 삽입 vs 단일 사용자 Won't 확정
- **CTO**: nullable owner_id 지금 삽입(비용 ~0, 역방향 소급은 26테이블 리라이트 규모). severity medium 유지.
- **Security**: 컬럼만 넣고 검증 없이 방치하면 '보호되고 있다'는 착시만 생성 — 컬럼은 통제가 아님. Won't 확정도 YAGNI상 정당. 핵심은 Open question을 닫는 것.
- **UX**: 컬럼은 싼 보험이나 멀티유저 전환의 실비용은 승인 워크플로·권한 화면 전면 재설계라 '리라이트 회피'는 과장. Open question을 명시적으로 닫는 것이 더 중요.
- → 세 명 모두 "결정 유예 상태 해소"에는 동의하나 **어느 방향으로 닫을지**는 사용자 몫.

### 4-4. 4축 점수 이중 저장의 처리 방식
- **CTO**: 정본 1곳 확정 + 동기화 계약 정의(드리프트 필연 방지).
- **UX**: 중복은 결함이 아니라 요구사항일 수 있음 — '선택 당시 점수'(감사 스냅샷)와 '현재 점수'(stale 반영)는 둘 다 필요하며 Company Memory 복기도 전자를 요구. '스냅샷 vs 라이브' 의미론 명시가 올바른 해결.
- **Security**: 전면 정규화는 과설계 — Risk/compliance 관련 필드만 정본을 못 박으면 충분(오래된 Risk 점수로 승인 판단하는 오염만 차단).
- (참고: 제목 후보 이중화·SNS 이중 모델링은 "변명의 여지 없는 미결정"으로 전원 동의 — 이 쟁점은 4축 점수에 한정됨)

### 4-5. 칸반 예외 상태(on_hold/stale/rejected)의 노출 방식
- **CTO**: 6번째 칼럼은 '5초 판단' 훼손, 숨기면 재검토 경로 유실 — 필터/별도 목록 뷰 절충 제안. 문서 수정이 아닌 제품 결정.
- **Security**: rejected/stale이 시야 밖에서 잊히면 가격 stale 콘텐츠가 그대로 수동 게시될 위험(가격 기준일 게이트의 사각지대). 단 매핑 작성은 상태 머신 정본 확정 이후여야 함.
- **UX**: 원지적자 — 매핑 테이블 부재 자체가 문제이며 노출 방식은 열려 있음.

### 4-6. P2-S1(URL 붙여넣기 상품 등록) 기능 존속 여부
- **CTO**: SSRF를 봉합해도 Naver의 봇 차단으로 실전에서 안정 동작하지 않을 가능성 높음 — 수동 입력 폼 유지 또는 공식 커머스 API로 **기능 스코프 자체를 재결정**해야 하는 기획 사안.
- **Security**: allowlist+내부 IP 차단+리다이렉트 제한을 조건으로 기술적 봉합 가능하나 ToS 자기모순은 별도 해소 필요.
- **UX**: 이 기능은 '입력 최소화' 1원칙의 상징 — 죽이더라도 수동 폼 폴백 UX를 먼저 정의해야 함.

### 4-7. raw_items 제3자 콘텐츠의 저장 범위와 법적 검토 스코프
- **Security**: 타인 블로그 본문 수집·저장의 저작권/개인정보 근거 미검토는 결함.
- **CTO**: 7일 TTL로 통상 관행 범위 — TTL 집행 여부 확인과 '발췌+URL만 저장'으로 대부분 완화되며, 정식 법률 검토는 TRD 범위 밖 사업 결정(기획 결함으로 계상은 스코프 오류).
- **UX**: '근거 문장 노출'이 제3자 저작물 상시 표시가 될 수 있어 출처 표기·인용 범위·원문 삭제 시 처리를 UX 스펙으로 정의 필요. 단 요약만 저장하면 '근거 원문 확인' 신뢰 장치가 약해지는 트레이드오프.

### 4-8. Prompt Injection 대응 수위
- **Security**: 외부 콘텐츠→프롬프트 주입, LLM 검수 우회, Company Memory 영구 오염(persistent injection)까지 신뢰 경계 설계 필요.
- **CTO**: 수동 게시 원칙의 인간 최종 게이트가 이미 최강 완화책 — 실질 피해는 '나쁜 추천' 수준. zod 스키마 검증 + 규칙 기반 override 두 가지가 최소선이고 신뢰 경계 프레임워크 전체 설계는 MVP 과잉.
- **UX**: 근거 노출 설계가 injection 완충재 역할. 단 규칙 기반을 최종 권위로 두면 오탐 증가 → [무시] 남용으로 귀결될 수 있어 검수 이중화와 [무시] 마찰 설계는 한 세트여야 함.
- → 최소선(§2-12)은 합의됐으나 **그 이상의 투자 수위**는 사용자 결정.

## 5. 리뷰어별 상세

### 5.1 CTO (초기 5 → 4)

**강점 평가**: Adapter 인터페이스 뒤의 AI 모델 격리 + cost_logs 1일차 추적, 예외 포함 상태 전체 열거와 테이블별 보존 정책, 자동 발행 금지로 플랫폼 리스크를 스코프 컷, 통일 API envelope + Naver 장애 폴백, 단일 사용자에 맞는 현실적 스택.

**핵심 지적 (high)**: ① 동기 파이프라인 vs Vercel 타임아웃(잡 큐·재개 전략 부재, tasks에도 없음) ② 상태 머신 소유자 불명확(String 4분산, 전이 검증·이력 없음) ③ 동일 데이터 이중 소스 3종(점수/제목/SNS) ④ cron 무인증+멱등성 부재+인증 스토리 상충. (medium) 커넥션 풀링·Vercel 플랜 미확정, 테넌시 부채, Company Memory 알고리즘 부재, API 일관성·명칭 불일치, <10초 성능 목표의 게이트 전파, 비용 서킷브레이커 부재. (low) Draft god-table + 4단 조인 체인.

**교차검증 태도**: UX 14건 중 agree 8·partial 4, Security 10건 전반 agree — 타 지적을 대부분 수용하되 **비용-효과 재계산**으로 정제(터치 타깃은 high 과대, Risk 의미론 변경 반대, a11y 전체 P1 반대, 저작권 법률 검토는 스코프 오류). 승인 워크플로는 "화면 삭제"라는 역방향 해법 제시. 서킷브레이커와 비동기 아키텍처는 "하나의 결정"이라는 통합 프레임 제공.

### 5.2 UX (초기 5 → 4)

**강점 평가**: 기획 단계의 구체적 접근성 명세(WCAG AA·4.5:1·aria), 검수 실패 회복 루프, Naver 장애 폴백 UX, '하루 30분·입력 최소화' 원칙의 일관 관철, 예외 상태 포함 상태 머신, 전 화면 근거 노출 설계.

**핵심 지적 (high)**: ① 승인 개념 화면-상태머신 단절([반려] 전이 미정의) ② 터치 타깃 자기모순(44px vs 28~32px) ③ IA 20여 메뉴 vs 화면 5개 ④ 헤더 상태 색상 의미론 붕괴+개수 불일치+우선순위 부재. (medium) 에러 토스트 4초 소멸, 점수 표기 붕괴, 칸반 매핑 부재, 빈 상태/온보딩 부재, 탭 명칭 충돌+편집 진입점 유실, [무시] 무마찰, 모바일 우측 패널 미정의. (low) 자동저장 혼합 멘탈 모델, 진입점 3중화, 성과 기록 리마인드 부재.

**교차검증 태도**: CTO·Security 지적 대부분 agree하되 **UX 결과로 번역해 보강** — 동기 파이프라인은 "대기 화면 설계 불가능 상태", 거짓 성능 목표는 "대기 UX 설계를 면제시킨 것", 비용 차단은 "차단된 상태의 화면 신설 필요". 고유 기여: 멱등키 '일자 unique'가 [새로 스캔] UX를 죽인다는 정정, 점수 중복의 스냅샷/라이브 재해석, lockout 자기잠금 경고, PaperclipDecision은 provenance 원료이므로 단순화 대상에서 제외.

### 5.3 Security (초기 4 → 3.5)

**강점 평가**: Compliance Gate의 Export 차단 게이트화, 자동 발행 금지의 구조적 ToS 회피, 고위험 카테고리 사전 차단, 401 acceptance 명시, env Secrets/HTTPS/request_id 로깅, 테이블별 보존 정책.

**핵심 지적 (high)**: ① cron 무인증(비용 DoS, acceptance 누락 라인까지 특정) ② `/api/products/import` SSRF + Won't 자기모순 ③ Stored XSS(sanitization/CSP 0건). (medium) Prompt injection(LLM 검수 스태킹), 인증 모순+CSRF/해싱/rate limit 미정, request_body 로깅, Supabase 권한 모델 미정의, 비용 상한 부재, 입력 검증 표준 부재, TRD §10 6줄+보안 게이트 0개. (low) 테넌시 IDOR, 보존 정책 모순+제3자 데이터 권리.

**교차검증 태도**: CTO 지적을 **보안 렌즈로 격상** — 상태 머신 String 분산은 Compliance Gate 우회 경로, 명칭 불일치(policy_rules)는 빈 룰셋 fail-open 위험, Company Memory 자유 텍스트는 persistent injection 경로, 승인 부재는 "유일한 인적 통제가 화면 장식". UX 지적에는 경보 피로=보안 운영 실패 모드, [무시] 무기록은 "이 리스트에서 가장 동의"로 강한 동조. 자기 지적에도 과잉 처방 경계(1인 도구에 풀 위협모델은 세리머니, 숨김>스텁, 수동 기록 유지가 ToS상 올바름).
